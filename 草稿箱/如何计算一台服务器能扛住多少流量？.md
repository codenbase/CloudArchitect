## 实验一：海量用户同时登录

这是一场对 CPU 算力的考验。在登录场景下，拖垮服务器的往往不是查数据库，而是复杂的数学运算。

登录场景，是纯粹的**计算密集型**任务。适用于所有“安全接入”的场景，如物联网设备接入，底层技术栈高度一致。

**登录性能消耗点：**
1. HTTPS（TLS 握手）：建立安全通道的非对称加密计算。
2. 密码哈希比对：使用 bcrypt、Argon2 或 PBKDF2 等慢速哈希算法，对密码加盐哈希，并与数据库密文比对。
3. 令牌签发：生成 SHA256 签名，JWT 序列化。

**压力测试（可能的瓶颈）：**
1. 第一波冲击：TCP/IP 与 TLS 协议栈的算力消耗，多核 CPU 的使用率开始急剧攀升。
2. 第二波冲击：慢哈希（bcrypt 算法）的“算力陷阱”，假设配置的 `cost` 参数为 10，单次哈希耗时可能达到 50 毫秒，一台 4 核服务器，每秒最多只能提供 4000 毫秒的 CPU 时间，那么这台机器在物理极限下，每秒最多只能处理 80 个登录请求（QPS = 80）。
3. 最终崩溃形态：排队与上下文切换灾难：当并发量达到 500 时，CPU 只有 4 个核心，剩下的 496 个请求去哪了？它们只能在 Go 的协程队列里排队等待。
	1. **现象 1：CPU 飙红。** `top` 命令下，`us`（用户态 CPU）死死卡在 100%，系统平均负载（Load Average）飙升到机器核心数的数倍甚至数十倍。
	2. **现象 2：RT 雪崩。** 吞吐量（QPS）不再增加，但 P99 响应时间从正常的 50ms 暴涨到 5000ms 甚至触发网关超时（504 Gateway Timeout）。
	3. **现象 3：数据库“闲得发慌”。** 此时查看 MySQL 状态，你会发现连接数很少，磁盘 I/O 几乎为零。因为应用层根本没有足够的算力把请求组装成 SQL 发给数据库。

**QPS 的价值和意义**
- **容量规划的“度量衡”**：精准计算出需要新购多少台服务器？
	- QPS 是一个瞬时指标，需要和日活用户（DAU）配合计算
	- “二八定律”容量评估模型：80% 的日常流量，集中在 20% 的时间里（即每天约 4.8 小时的高峰期）。
	- 66 QPS 总承载量：66 次/秒 × 17,280 秒（4.8 小时） ≈ **114 万次登录**。
	- 全天总承载量：114 万 ÷ 80% ≈ **142 万次登录**。
	- 强登录属性系统（如企业OA打卡系统）理论上足以支撑 140 万 DAU（日活）的盘子。
	- 弱登录属性系统（如内容社区/电商）理论上足以支撑上千万的 DAU。
	- 致命的例外（业务突发风暴）：“秒杀”或者“大V发红包”，10 万人会在 **1 秒钟**内同时涌入。这时候你的瞬时峰值需求是 100,000 QPS。
	- 这就是为什么计算 QPS 必须结合**业务流量模型**。
	- 当 QPS 达到多少时，需要扩容？黄金扩容红线：70% 法则。实际观测到的 QPS 经常性地触达你压测极限的 **70%**。
	- 
- **架构演进的“发令枪”**：
	- 百级 QPS：单机随便跑
	- 万级 QPS：必须引入 Redis 缓存，消息队列削峰，读写分离
	- 十万/百万级 QPS：必须走向微服务拆分、分库分表、多活机房。
- **性能优化的“验钞机”**：当你优化系统之后，如果没有 QPS 对比（66 vs. 260），你向领导汇报只能说“系统变快了”，这毫无说服力。有了 QPS，你可以理直气壮地说：“通过调整算法，我们将单机计算吞吐量提升了 400%（4倍）”。
- **系统监控的“心电图”**：生产环境某个接口的 QPS 常年稳定在 2000，今天突然掉到了 50，你就知道出事了！
- **服务等级协议的“契约”**：对于开放 API 平台，QPS 是写进商业合同的，比如本接口承诺提供单租户最高 500 QPS 的并发能力，超出部分将限流。

**调优方法：**
1. 协议层优化：复用与降级。 开启 HTTP Keep-Alive & TLS 卸载（需引入网关）。
2. 应用层优化：哈希成本与并行度。动态调整 Hash Cost （牺牲一点安全性）& 优化并发模型（Go 合理设置 `GOMAXPROCS`）。
3. 业务优化：黑名单缓存限制高频非法登录 & 增加 JWT 令牌有效时间（有效期免登录）。
4. 强力限流（Rate Limiting）阻断 DDoS。
5. 引入前置的风控验证码，图形验证码或滑块验证。

**慎用免登录场景（安全性和性能永远是互斥的）：**
1. 核心资产高危操作，如修改密码，银行卡相关，VPP 下发电网指令等。
2. 防重放攻击，云端向一台智能汽车发送“解锁车门”的指令。凭证决不能复用。
3. 完全无状态的 RESTful 开放 API。对于面向第三方开发者的 Open API 网关，为了极端的解耦，有时不维护任何会话状态。每一次 API 调用，调用方都必须使用 AppKey 和 AppSecret 对整个请求体进行高强度的 HMAC 签名。服务端每次都要重新计算整个 Body 的签名来比对。请求量越大，CPU 越惨。

## 实验二：高频浏览热门文章

某篇重磅文章全网刷屏，大量用户同时疯狂刷新详情页；或者在 AIoT 场景下，几十万台设备同时向网关拉取一份体积较大的全量配置文件。

动作：`GET`  -> 路由  ->  查数据库 -> 结构体组装 -> JSON 序列化 -> 客户端。
思考：为什么此时不要 JWT 验签？

1. 关系数据库的 Buffer Pool（内存缓冲池）使得同一篇文章只有第一个请求会触发物理磁盘的读，随后都是从内存读取，不会造成磁盘压力。
2. 网络带宽能够承载的 QPS 是多少？
	1. 假设 千兆（1 Gbps），理论极限是 $125 \text{ MB/s}$。
	2. 极限 QPS 估算：$125,000 \text{ KB/s} \div 50 \text{ KB/request} = 2500 \text{ QPS}$。
3. TCP/IP 协议栈与内核资源的耗尽。文件描述符打满，TIME_WAIT 风暴。
4. JSON 序列化引发的内存与 GC 灾难。

**IDE 提示词**
在当前的 `monolith-stress-lab` 项目中新增『实验二：高频浏览热门文章』的压测场景。请求直达业务逻辑，从数据库直接拿出数据，返回给客户前端。这个实验要能够测试两种子场景：1. 文章内容很大；2. 文章内容很小。先给出一个计划，由我确认。

**BPS (Bytes Per Second, 字节吞吐量)** 和 **PPS (Packets Per Second, 每秒发包数)**。
1. BPS = QPS × 一个请求的网络包大小
2. PPS = QPS × 一个请求的网络包数量
3. 带宽（Bandwidth）是物理天花板（固定值），比如 1Gbps 的网卡，它决定了这条高速公路最多能修几根车道。 BPS 和 PPS 是当下的实际车流量（动态值），它们永远不可能超过物理天花板。

这考验的是I/O与网络带宽的承载能力。
1. 文章内容很大：网络带宽瞬间打满？
	1. 开启 Gzip/Brotli 压缩
	2. CDN（内容分发网络）
	3. 死于 BPS
2. 文章内容很小：Web服务疯狂序列化？内存压力飙升？数据库连接池被耗尽？
	1. 引入 Redis 缓存
	2. 死于 PPS 
	3. 对抗海量 PPS 最有效的手段：长连接复用（HTTP/2 或 WebSocket）和 eBPF（让网络包直接绕过操作系统的内核 TCP/IP 协议栈，直接送达用户态内存）
	4. 数据库连接池大小设置为 CPU 核数的 2-4倍。

**PG 数据库连接池问题**
在“小文件极速读取”场景下（200 VUs 压测并发），将 Go 客户端的数据库连接池（`MaxOpenConns`）设置为 100 时，系统不仅吞吐量未达物理极限，反而出现了约 0.5% 的极少量 `Failed` 请求。这些请求并非死于网络传输或系统长队超时，而是瞬间“暴毙”。
通过追踪到底层日志（`FATAL: remaining connection slots are reserved for roles with the SUPERUSER attribute`），确诊了报错的真正元凶：**客户端并发建连突破了服务端的配额红线。**
- **PostgreSQL 的硬性限制：** PG 默认的最大连接数 `max_connections` 为 100。
- **权限预留机制：** 为了保证在极度拥堵时 DBA 依然能登录数据库进行抢修，PG 默认设置了 `superuser_reserved_connections = 3`。这意味着留给普通业务账号（如 `stressuser`）的实际可用连接数**仅为 97 个**。
- **踩踏过程：** 高并发瞬间，Go 程序试图打满配置的 100 个连接。前 97 个连接顺利建立，而尝试建立最后 3 个连接时，触发了 PG 的权限防御机制被瞬间拒载。这 0.5% 的失败率，正是由于少部分请求不幸命中了这 3 个“坏死”的连接通道所致。
要解决此问题，必须保证：**客户端应用集群的最大连接数总和 $\le$ 数据库服务端最大连接数 - 预留连接数。**
- **解法 1：纵向扩容服务端（治标）。** 修改 PG 的 `postgresql.conf`，通过 `ALTER SYSTEM SET max_connections = '500'` 并重启服务。此时普通用户可用额度变为 497。
    
- **解法 2：横向收缩客户端（治本，强烈推荐）。** 将 Go 程序的 `MaxOpenConns` 降至安全水位（如 15 或 30）。

**延伸思考：连接池到底设置多大合适？**
新手常有一种错觉：“在机器资源充足时，连接池越大并发越强”。**这在关系型数据库架构中是致命的错误。**
1. **连接池并非越大越好（Small is Fast）：** 数据库连接池的大小应严格匹配**数据库服务器的物理 CPU 核心数**，而非前端的并发量。业界标准公式为 `(Core_Count * 2) + 1`。过大的连接池会引发严重的操作系统上下文切换（Context Switch）和内核态锁竞争，导致系统抖动（Thrashing），整体吞吐量反而大幅下降。
2. **高并发下的漏斗防御模型：**
    
    - **用户态排队（最优解）：** 宁愿让海量请求在 Go 程序的内存中（基于极轻量级的 Goroutine）排队等待空闲连接，也不要让大量请求涌入内核态或数据库进程中排队。限制连接池大小（如 15），本质上是利用漏斗机制保护脆弱的下游物理资源。
        
    - **网关层限流（终极防御）：** 当并发量大到连内存（用户态队列）都面临 OOM 风险，或者排队时间必然超过业务 SLA 忍耐极限时，必须在更上层的 API 网关处果断采取限流（Rate Limiting）或熔断措施，快速失败（Fail-Fast），以保全系统的整体存活。

**压测方法论**：寻找拐点，步增模式（Ramping VUs，比如从 10 慢慢增加到 200）。

**计算方法论**
1. 千兆网卡，QPS，每秒请求数据量，网络流速 之间的关系： $3099 \text{ QPS} \times 50 \text{ KB/请求} \approx 154,950 \text{ KB/s} \approx 151 \text{ MB/s}$。
2. 利特尔法则（$并发数 = QPS \times 响应时间$），在 QPS 被网络带宽死死卡住不变的情况下，并发数增加，响应时间变慢。

**什么时候才“必须”优化？**
1. 业务预期爆发（天花板危机）预估峰值将达到 5 万 QPS，而压测只有 9000。
2. SLA 体验跌破底线（长尾危机）P99 必须控制在 20ms 以内，而压测是 92ms。
3. IT 成本极度超标（降本增效）为了维持这个 QPS，用了 100 台极其昂贵的顶配云服务器。老板要求下个季度削减 30% 的 IT 预算，此时的优化，是为了“用更少的机器跑出同样的 QPS”**。

干货：用 `dstat` 或 `nload` 看网络带宽，用 `netstat` 看 TCP 连接数。

## 实验四：内存泄漏与 GC 恐慌

在真实的业务中，最让人头疼的往往不是 CPU 打满，而是 **OOM（Out of Memory）** 和 **垃圾回收（GC）停顿**。

**现实价值：** 在处理海量终端状态缓存、或者解析极度复杂的底层协议帧时，内存管理是决定系统存活时间的核心命门。

## 压测的意义

1. 容量摸底（Capacity Planning）：建立底气，为了知道一台服务器能装多少人？
2. 探寻短板：指明优化方向，如登录接口的短板是 CPU，大文件的短板是网卡带宽，高频短连接的短板是数据库连接池。
3. 防御策略设定（Defensive Governance）：画下红线。当你知道单机的极限物理天花板是 9400 QPS 后，压测的任务就结束了。接下来的动作不是改代码，而是去 **API 网关上配置限流（Rate Limit）规则**。
