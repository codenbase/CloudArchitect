## 压测的意义

1. 容量摸底（Capacity Planning）：建立底气，为了知道一台服务器能装多少人？
2. 探寻短板：指明优化方向，如登录接口的短板是 CPU，大文件的短板是网卡带宽，高频短连接的短板是数据库连接池。
3. 防御策略设定（Defensive Governance）：画下红线。当你知道单机的极限物理天花板是 9400 QPS 后，压测的任务就结束了。接下来的动作不是改代码，而是去 **API 网关上配置限流（Rate Limit）规则**。

## 压测方法论

压测的目的是为了验证系统能否在目标 QPS 和红线 SLA 之上健康存活。

**第一步：明确业务目标**

为了计算出目标 QPS，我们必须先和产品运营人员对齐：预估的最大 DAU 和同时在线人数是多少？这能让我们算出下面两个信息：

1. 从 DAU 算出“日常峰值 QPS”
2. 从“同时在线人数”算出“极限并发 QPS”

**DAU（Daily Active Users，日均活跃用户数）** 代表一天内有多少独立用户访问。 但用户不是均匀访问的，互联网极其经典的流量规律是 **二八法则**（80% 的请求集中在 20% 的时间里，比如早高峰和晚高峰）。我们假设 DAU 是 **100 万**，每个用户每天平均打开 APP **10 次**。那么，全天总请求量就是 1000 万次 API 请求，根据二八法则，将 80% 的请求量（800 万）压缩在 20% 的时间（4.8 小时 = 17280 秒）内。日常峰值 QPS 就是：`800 万` 除以 `17280 秒` 约等于 `462 QPS`。

**同时在线人数** 代表在某一瞬间，有多少人正拿着手机停留在你的 APP 页面上。假设某个大 V 爆出了一个“惊天大瓜”，引发 10 万人同时围观，每隔 5 秒钟刷新、点赞、评论。此时的极限 QPS 就是：`10 万` 除以 `5 秒` 等于 `20000 QPS`。

除了 QPS，还有 SLA 也非常重要。

SLA（Service Level Agreement，服务等级协议）必须要满足一个红线，例如 P99 延迟必须小于 200ms，错误率小于 0.01%。这不是老板拍脑袋定出来的，而是基于一个商业共识：网页加载每慢 1 秒，就会流失 10% 的转化率（钱）。

用户看到首屏网页的忍耐极限通常是 1~2 秒，扣除掉 DNS 解析、TCP 握手、网络传输，再扣除前端页面的渲染时间，留给后端服务器的时间可能只剩下不到 200ms。这就是为什么绝大多数核心业务接口的 P99 SLA 红线，都被死死定在 **200ms 以内**的原因。

另外，在微服务架构中，由于调用链（`网关 -> 服务 A -> 服务 B -> 数据库`）的存在，每个服务扣除自己的计算时间后，留给下游服务的 SLA 可能会更低。

为什么要容许 0.01% 的错误率呢？这是一个 ROI（投资回报率）问题。要知道，100% 不报错的系统是不存在的，如果把可用性提升到 99.999%（金融级），可能需要做异地多活、专线双网，成本暴增 100 倍（花 100 万块）。所以，**容忍 0.01% 的错误率，是技术向商业妥协的产物**，为了那极端的 0.01% 去投入海量服务器，ROI（投资回报率）是极低的。

**第二步：设计压测场景**

我会分三个阶段设计压测场景，每个阶段的目的各不相同：

1. **基准测试（Benchmark）：** 单接口、空数据或极少数据、极限并发。目的是采用“控制变量法”，寻找单点物理极限，这可以在没有任何干扰的情况下，摸清一个接口的最快响应时间和 CPU 消耗基线。
2. **容量测试（Load Test）：** 目的是评估当前系统架构在真实业务中的最大“承载力”。使用千万级铺底数据和真实的线上流量比例（比如 80% 读文章，15% 点赞，5% 发评论），采用“步增模式”（Ramping VUs，比如从 10 慢慢增加到 200），像爬楼梯一样，阶梯式增加并发，寻找系统 QPS 停止增长、响应延迟开始突增的那个**黄金拐点**。
3. **稳定性测试（Soak Test/疲劳测试）：** 在系统最大承载量的 80% 水位下，连续施压 24 小时或 72 小时。这是专门用来揪出内存泄漏（OOM）、Goroutine 积压和数据库慢查询的终极武器。

**第三步：建立监控体系**

1. k6 的压测结果：看 QPS、P99 延迟。
2. API 服务应用：HTTP 错误码分布（区分是 500 还是 504）
3. **运行时环境：** 看 Go 的 GC 停顿时间、Goroutine 数量、内存分配速率。
4. **服务器：** 盯着 `top` 和 `iostat`，看是 CPU 的 `sy`（系统态）打满了，还是网卡的 `BPS`（物理带宽）撞墙了，或者是磁盘的 `wa`（I/O 等待）飙升了。
5. **中间件：** 严密监控数据库连接池的活跃度、排队深度，以及是否有 `FATAL: too many clients` 等底层的权限或配额阻断。”

**第四步：问题定位与调优**

发现性能不达标，才是架构师真正发力的时候。

如果是吞吐量上不去但 CPU 很闲，我会优先排查阻塞等待（比如数据库连接池太小导致排队，或者下游第三方接口超时）。

就像在处理高并发短连接查询时，初学者可能会盲目调大数据库连接池，但这往往会导致严重的 CPU 上下文切换内耗。专业的做法是基于数据库物理核心数来限制连接池大小（比如设为 15），通过在应用层构建漏斗机制进行排队，以极小的并发度实现对数据库的高效复用，最终彻底消灭超时报错，提升全局吞吐量。

每一次调优后，必须重新执行同样的压测场景，直到性能指标完全契合第一步设定的业务 SLA。

### 给后端兄弟的真实压测指南

很多开发兄弟一接到“压测”任务，第一反应就是打开工具，填个高并发数，对着接口一顿狂轰滥炸。

但说实话，专业的压测绝不是单纯为了“压垮系统”。压测的真正目的，是验证我们的系统，能不能在**目标并发量**和**不可退让的延迟红线**之上，健康地活下来。

想要做到这一点，主要分四步走。

**第一步：别急着敲代码，先跟产品经理“算算账”**

压测前，我们得先搞清楚真实的目标 QPS 是多少。这时候你要去找产品或运营对齐两个核心数据：**DAU（日活）** 和 **同时在线人数**。

举个例子，假设你们的 APP 日活是 100 万，每个用户每天平均打开 10 次。全天总请求量就是 1000 万次。

根据互联网经典的“二八法则”，80% 的流量会集中在全天 20% 的时间里（比如早晚高峰）。简单算一下，把 800 万次请求压缩在大概 4.8 小时里，**日常峰值其实也就不到 500 QPS。** 看，100 万日活听着吓人，日常压力并不大。

但如果是突发热点呢？

比如某大 V 爆了个惊天大瓜，瞬间涌入 10 万人同时在线围观。假设大家每隔 5 秒钟疯狂刷新、吃瓜点赞，此时的**极限 QPS 直接飙升到 20000！** 这才是我们要死磕的压测目标。

除了算 QPS，我们还要定一条叫 **SLA（服务等级协议）的红线**。比如：接口响应时间必须小于 200 毫秒，错误率不能超过 0.01%。

这绝对不是老板拍脑袋定的。商业世界有个共识：网页加载每慢 1 秒，就会流失 10% 的钱。用户点开页面的忍耐极限最多就 1 到 2 秒，扣除掉网络传输、前端渲染这些损耗，留给咱们后端服务器的时间，往往就只剩下区区 200 毫秒了。如果还要经过网关、各种微服务和数据库的层层调用，留给最底层服务的响应时间可能只有几十毫秒。

至于为什么允许 0.01% 的错误率？因为想要达到 100% 不报错的“金融级”纯净系统，可能需要花 100 倍的钱去搞异地多活和专线。技术永远要向商业的投资回报率（ROI）妥协，偶尔弹个“网络开小差”，性价比往往是最高的。

**第二步：像爬楼梯一样设计压测场景**

弄清了目标，就可以开始设计压测场景了。我一般会循序渐进地过三关：

首先是**基准测试**。这就像在无风的实验室里测汽车发动机。我们用空数据或者极少的数据，单拉出一个接口跑极限，剥离一切干扰，纯粹摸清这个接口最快的响应速度和 CPU 消耗底线。

其次是**容量测试**。把发动机装进车里去跑烂路。我们会往数据库里塞入千万级的真实“铺底数据”，并按照线上真实的流量比例（比如 80% 读文章、15% 点赞、5% 评论）混合施压。并发数要像爬楼梯一样一点点往上加，直到你发现系统的 QPS 涨不上去了，或者延迟突然翻倍了——恭喜你，你找到了系统的“黄金承载拐点”。

最后是**疲劳测试**。在系统最大承载量的 80% 水位下，连续轰炸它 24 小时甚至 72 小时。很多隐藏极深的致命慢性病（比如内存泄漏、协程积压），只有在这个阶段才会彻底暴露。

**第三步：布下天罗地网的监控**

压测跑起来的时候，只盯着压测工具的面板看 QPS 是远远不够的。你得自上而下盯紧整个盘子：

看最外层的 HTTP 错误码，分清到底是 500（内部报错）还是 504（超时）；看 Go 语言底层的 GC 垃圾回收有没有疯狂停顿；盯着服务器的监控，看是 CPU 系统态打满了，还是网卡的物理带宽撞墙了；最后，千万别忘了看中间件，数据库的连接池是不是在排长队，有没有触发底层的连接数阻断。

**第四步：别乱动代码，揪出真凶再调优**

跑出不达标的数据，才是架构师真正发力的时候。

如果发现系统吞吐量死活上不去，但 CPU 却闲得很，千万别急着加机器。这大概率是哪里的**阻塞等待**出了问题。

最典型的误区就是高并发查询时，新手喜欢盲目调大数据库连接池，以为连接越多越快。但这反而会让有限的 CPU 陷入疯狂的“上下文切换”内耗中。真正的老鸟，会根据数据库的物理核心数，把连接池死死限制在一个很小的值（比如 15）。宁愿让多出来的请求在极其轻量的 Go 内存里悄悄排队，也绝不让海量请求涌向底层的数据库。通过这种“漏斗机制”，反而能彻底消灭超时报错，让全局吞吐量翻倍。

记住，每一次调整配置或代码后，必须用同样的场景重新压测，直到性能完美契合我们第一步算出来的业务红线。
