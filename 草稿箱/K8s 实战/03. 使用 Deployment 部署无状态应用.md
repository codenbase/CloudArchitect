
1. Pod 没有自动化管理机制，如故障自动恢复、水平扩展等。kubernetes 是通过工作负载的逻辑概念进行 Pod 的全生命周期自动化管理。
2. kubernetes 内置的工作负载有 Deployment、ReplicaSet、ReplicationController、DaemonSet、StatefulSet、Job、CronJob 等。
3. 随着 Kubernetes 面对的业务复杂度指数级上升，早期的核心工作负载 `ReplicationController (RC)` 逐渐被 `ReplicaSet (RS) + Deployment` 的组合取代。RC 的淘汰源于两个底层架构设计的死穴：
	- **数据维度**：简陋的基于等式（Equality-based）的标签选择器。
	- **控制维度**：双层架构（RC -> Pod）导致状态机管控能力极度受限。
4. 为了在不破坏 API 向后兼容性的前提下解决这些痛点，K8s 官方放弃了在 RC 上直接打补丁，而是进行了彻底的架构重构，确立了 `Deployment -> ReplicaSet -> Pod` 的三层控制链路。
5. **数据维度的破局：从“键值匹配”到“关系查询”** ：RC 基于等式的标签选择策略（如 `app=web`），在面对复杂的灰度发布、高可用打散调度以及第三方组件生态时显得捉襟见肘。新的 RS 引入了强大的基于集合（Set-based）的匹配机制（如 `version in (v1, v2)` 或 `NotIn`）。这本质上是将 K8s 的标签过滤系统，从简单的“键值对匹配器”升级成了一门“类 SQL 的关系型查询语言”，补齐了现代分布式系统治理中不可或缺的 `OR` 和 `NOT` 逻辑拼图，实现了对复杂业务群体的精准“框选”与“隔离”。
6. **双层架构的局限性** ：在双层架构（RC -> Pod）下，进行安全的滚动更新和回滚极其困难：
	- **服务端实现的灾难**：若强行在 RC 内部实现滚动更新，RC 必须亲自维护极其复杂的并发状态机（记录已删减和新增的 Pod 数量），还要引入底层数据库来存储历史配置以支持回滚。这会导致 RC 的代码逻辑极度耦合和臃肿，沦为极易产生 Bug 的“上帝对象（God Object）”。
	- **客户端驱动的隐患**：为了避免服务端臃肿，早期的滚动更新被迫采用客户端驱动（`kubectl rolling-update`）。但这带来了致命的稳定性风险：一旦更新中途发生执行端断网、死机或人为误操作，集群的自动调谐循环会被强行中断，导致系统永远卡在新老版本共存的“半死不活”中间态。
7. **三层架构的优雅** ：`Deployment -> ReplicaSet -> Pod` 的三层架构，通过极致的**单一职责解耦**，完美化解了滚动更新与回滚的难题：
	- Deployment 完全脱离了底层的 Pod 启停脏活。在滚动更新时，它只做一件事：创建新的 RS，并像“拧收音机旋钮”一样，把旧 RS 的期望副本数往下拧、把新 RS 的期望副本数往上拧。
	- RS 只负责死死盯住当前分配给自己的目标数量，处理节点宕机等并发异常，确保最终一致性。
	- 当更新完成后，旧的 RS 虽然副本数降为 0，但其作为 API 资源的实体依然驻留在集群中，充当完美的“历史快照”。发生故障需要回滚时，Deployment 只需将旧 RS 的旋钮重新拧回目标值，即可在秒级安全恢复业务。
8. 大多数情况下，Deployment 不关心 Pod 被调度到哪个 Node（完全交由调度器基于 CPU/内存 的默认算法打散）。但当业务对底层物理拓扑有特殊要求时，K8s 提供了三种由简入深的干预手段：
	- **NodeSelector（节点选择器 - 绝对命令）**：最原始的标签匹配。例如强制将 MySQL 调度到带有 `disk=ssd` 标签的节点。缺点是过于死板，一旦目标节点资源不足或宕机，Pod 就会永远卡在 Pending（挂起）状态。
	- **NodeAffinity（节点亲和性 - 弹性谈判）**：NodeSelector 的全面升级版。它不仅支持集合语法（In/NotIn），最强大的是引入了“软硬限制”策略。你可以规定：“我**优先**想要 SSD 节点（软限制），但如果实在没有，普通机械硬盘的节点也能凑合，千万别让我起不来”。
	- **PodAffinity / PodAntiAffinity（Pod 亲/反亲和性 - 社交拓扑）**：前两者都是“Pod 找 Node”，而这一层是“Pod 找 Pod”。
		- **PodAffinity（亲和）**：解决强耦合服务的“物理同驻”需求，例如业务 Pod 必须和它的专属缓存 Pod 部署在同一台物理机上，以追求极致的本地网络极低延迟。
		- **PodAntiAffinity（反亲和）**：解决高可用的“物理隔离”需求，例如同一个核心服务的 3 个副本，绝对不能（或尽量不要）被调度到同一台 Node（甚至同一个可用区/机房）上，防止单点物理故障导致服务全军覆没。
