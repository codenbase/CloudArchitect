**架构拆解：一次标准的动静分离**

如上图所示，一个现代网页的真实加载链路是这样的：浏览器首先从文件服务器拉取 HTML/CSS/JS 静态骨架，随后在渲染过程中，通过 JavaScript 发起网络请求，调用后端 API 获取真实数据。

为了完美契合这种客户端渲染（CSR）模式，我们将实验环境清晰地划分为以下四个核心组件：

- **统一流量入口 (Ingress Gateway)：** 负责根据 URL 路径进行智能路由分发。
    
- **前端静态文件服务 (web-demo-frontend)：** 基于 Nginx 部署，纯粹的“无情发牌机器”，只负责下发静态页面。
    
- **后端 API 服务 (web-demo-backend)：** 使用 Go 语言原生开发，处理真正的业务逻辑。
    
- **数据库服务 (postgres)：** 负责数据的持久化存储。
    

**为什么要做这么“复杂”的拆分？**

## 部署方法论：谋定而后动

接下来的实操环节，你需要一个就绪的 Kubernetes 集群。如果你手头还没有，强烈建议回顾我之前更新的**《K8s 二进制部署实战课程》**，跟着教程在本地零成本拉起一个纯净的测试集群。

在敲下第一行部署命令之前，我们先在脑海里建立完整的全局观。云原生部署绝不是把配置文件一锅炖，它有着极其严谨的依赖层级。

为了保证实验环境的整洁，我们的第一步是划定一块“沙盒”——创建一个专属的 **Namespace (命名空间)**。本次实战的所有组件都将收敛于此。这么做最大的好处是**“阅后即焚”**：测试结束后，只需一条命令删除这个 Namespace，就能瞬间清理所有资源（注：CRD 等全局资源或保留策略的 PV 除外），绝不会给你的集群留下任何“垃圾”。

我们的第一步是创建一个名为 `web-demo` 的 **Namespace（命名空间）**。 这不仅仅是为了“测试完方便一键删除”那么简单，在企业级架构中，Namespace 是资源隔离的最小沙盒。它意味着：

- **网络隔离与访问控制：** 我们可以基于 Namespace 设定网络策略（Network Policies），防止其他业务线的 Pod 恶意访问我们的数据库。
    
- **资源配额管理：** 防止某个异常服务耗尽整个集群的 CPU 和内存（ResourceQuota）。

在这个沙盒里，我们将严格遵循**“先配置后计算，先底层后网络”**的架构法则，依次完成以下核心战役：

1. **备足粮草 (ConfigMap & Secret)**：配置先行。在启动任何容器之前，优先将数据库的敏感凭证（账号密码）和应用的基础配置安全地注入到集群中。
    
2. **夯实底座 (StatefulSet)**：部署系统的核心依赖——PostgreSQL 数据库。针对这类极其看重数据持久化和网络标识唯一性的“有状态服务”，我们将采用 StatefulSet 模式进行精准部署。
    
3. **拉起业务 (Deployment)**：当底层数据库和配置准备就绪，使用 Deployment 丝滑拉起我们无状态的 Web API 后端与 Nginx 前端页面。
    
4. **打通经脉 (Ingress)**：内部服务运转正常后，配置 Ingress 规则，作为整个集群的“七层大闸”，将外部的 HTTP 请求精准分发到对应的微服务。
    
5. **终极演进 (Helm 一键部署)**：当手工打造完所有的原生 YAML 后，我会带你拔高一个维度。我们将使用 Helm 把整套架构参数化，制作成企业级的 Chart 压缩包，真正实现“一行命令，全栈落地”。

10. 使用ConfigMap配置应用程序
	1. k8s 中有哪些配置方式？为什么用 ConfigMap？
	2. ConfigMap 包含多个描述配置信息或者文件的键值对。
	3. 通过文件或环境变量的方式呈现给Pod中的容器。
	4. ConfigMap 的局限性：
		1. 更改配置实际上不会触发对现有Pod的更新。配置仅在Pod重新启动时才会生效。这样会导致配置修改不是基于健康检查的，可能是临时的或随机的。
		2. ConfigMap只在版本控制系统中进行版本管理，执行回滚可能非常困难。
	5. 更好的方式是在ConfigMap的命名中加入版本号。
		1. 更改时，只需要创建一个新的v2版本，然后更新Deployment资源以使用该版本的ConfigMap，而不是直接更新现有的ConfigMap。
		2. Deployment将会自动触发重新部署，执行恰当的健康检查和服务停顿以切换配置。
		3. 如果需要回滚，由于v1的配置仍然保留在集群中，只需再次更新Deployment即可。


11. 使用Secret管理认证
	1. 在任何实际的应用程序中，我们都需要保护服务之间的连接。
	2. 为了确保用户及其数据的安全
	3. Kubernetes提供了内置的Secret资源来管理机密数据。
	4. Volume实际上是一个文件或目录，可以将用户指定的位置挂载到正在运行的容器中。对于Secret，Volume被创建为基于RAM的tmpfs文件系统，然后挂载到容器中。这样一来，即使机器被物理上入侵（在云中是不太可能的，但在数据中心是可能的），攻击者也很难获取到Secret。
	5. 默认情况下，Secret是未加密存储的。如果你想要加密存储Secret，可以集成密钥提供程序以获取一个密钥，Kubernetes将使用该密钥来加密集群中的所有Secret。
	6. 请注意，尽管这样可以保护密钥免受对etcd数据库的直接攻击的影响，但仍需确保Kubernetes API服务器的访问安全。
	7. CSI（Container Storage Interface，容器存储接口）驱动程序使你能够使用位于Kubernetes集群外部的KMS（Key ManagementSystem，密钥管理系统）。这通常是大型组织或受监管组织内合规性和安全性的要求。

14. 使用Helm将你的应用程序参数化
	1. helm，k8s  ，Helm Chart 中的 `values.yaml` 将镜像版本指向刚刚发布到 GHCR 的镜像版本，随后发布到 GitHub Pages。
	2. 为什么需要使用 Helm ？
		1. 需要把应用部署到多个不同的环境
	3. 在Helm中，一个应用程序被打包在一个称为chart的文件集合。
	4. 一个chart以chart.yaml 文件为开始，它定义了chart自身的元数据。
	5. 该文件位于chart目录的根路径下（例如，frontend/）。该目录中有一个templates子目录，其中放置着模板文件。模板基本上是前面示例中的一个YAML文件，只是其中某些值被替换为引用参数。
	6. 这意味着你可以在部署chart时使用合适的参数来替换副本数的值。这些参数被定义在values.yaml 文件中。应用程序的每个部署环境中都将有一个values.yaml 文件。
	7. 你会希望对应用程序进行参数化，以使其配置在不同环境中更具可重用性。像Helm这样的打包工具是这种参数化的最佳选择。
