
## ConfigMap

1. 一切皆容器的云原生时代，应用部署的最佳实践是先将应用打包成镜像，再分发至不同的环境。为了在开发、测试、生产环境中复用同一个版本的镜像，我们绝不能将配置文件“写死”并打包进镜像中。为此，Kubernetes 设计了 ConfigMap 资源对象来实现应用配置的统一管理。
2. ConfigMap 供容器使用的典型用法：
	- 生成容器内的环境变量，适合简单的配置项（如数据库连接地址、环境标识）。
	- 设置容器启动命令的命令行参数（需设置为环境变量）
	- 以 Volume 的形式挂载为容器的文件或目录，适合复杂的长文本配置（如自定义的 `nginx.conf` ）。
3. ConfigMap 必须在 Pod 之前创建，Pod 才能引用它。如果 ConfigMap 不存在，或者被 Pod 引用的 key 不存在，则 Pod 无法启动。但是，如果在 Pod 的配置中设置引用 ConfigMap 是可选的（optional），则 Pod 会被正常创建和启动，只是容器内引用的 ConfigMap 的环境变量的值或挂载的文件的内容会被设置为空。
4. 如果 Pod 使用 envFrom 基于 ConfigMap 定义环境变量，则无效的环境变量名称（例如以数字开头）将被忽略，并在事件中被记录为 “InvaildVariableNames”。
5. ConfigMap 受命名空间限制，只有相同的命名空间中的 Pod 才可以引用它。
6. ConfigMap 无法用于静态 Pod。
7. 创建 ConfigMap 的三种方式：
	- 基于本地配置文件
	- 基于 kubectl 命令行  `kubectl create configmap NAME --from-file=[key=]source`
	- 基于 kustomize
8. ConfigMap 存在局限性：
	- 直接更改 ConfigMap 资源，**不会**自动触发现有 Pod 的滚动更新。虽然你可以在应用程序内自己编写监听文件变化的逻辑（例如 Go 的 viper 包）来实现热更新，但强烈不推荐。因为这会绕过 K8s 的健康检查（探针 Probes）机制，假设热更新了一个错误的数据库连接串，连不上数据库，应用程序会疯狂给用户返回 HTTP 500 错误。因为在 K8s 眼里，这个 Pod 的“生命周期”根本没有发生改变，它依然是那个处于 `Running` 状态的旧 Pod，K8s 并没有触发“发布”流程，自然也不会用探针去拦截它。等探针察觉到不对劲把 Pod 踢下线时，错误已经暴露给真实用户了。也许你会说，那我就不实现热更新了，直接改线上的 ConfigMap，那更加可怕，你这个改错的配置很可能在 3 天后， Pod 发生一次意外的崩溃被 K8s 重新调度，因而读取了你 3 天前的错误配置，起不来了。而你一脸懵，为什么 3 天前改的配置今天凌晨引爆了系统。这种不可控的随机切换完全可以避免。
	- ConfigMap 会失忆。ConfigMap 无法做到像 Deployment 那样，修改后发现不对能够立马回滚。K8s 会为 Deployment 在底层保留上一个版本的记录（也就是 ReplicaSet），而 **不会保存** ConfigMap 的历史版本。因此，ConfigMap 回滚极其困难，
9. 为什么必须坚持“配置不可变、版本化发布”？
	- 为了解决这个痛点，我们需要引入“不可变基础设施”的思维：在 ConfigMap 的命名中加入版本号（例如 `web-config-v1`）。
	- 当配置发生变更时，**绝对不要**去修改现有的 `web-config-v1`。而是创建一个全新的 `web-config-v2`，随后修改 Deployment 的 YAML 文件，将引用的 ConfigMap 指向 v2。
	- 这种做法会立刻触发 Deployment 的滚动更新机制。K8s 会先启动挂载了 v2 配置的新 Pod，执行健康检查，确认无误后再优雅停机旧 Pod。如果发现 v2 配置有误引发了故障，由于集群中依然保留着完整的 `web-config-v1`，你只需一条命令即可将 Deployment 瞬间回滚。
	- 这就是 ConfigMap **标准的、安全的更新流程（利用版本化 ConfigMap + 触发 Deployment 滚动更新）**
10. K8s 官方显然也深刻意识到了“原位修改配置”带来的灾难。因此，从 Kubernetes v1.19 版本开始，为 ConfigMap 新增了 immutable 字段，用于设置配置数据不可修改，如需修改，只能通过先删除再重建来实现。这个特性可以完美地配合实现我们的版本化 ConfigMap。这么做的好处，不仅消除了意外更新 ConfigMap 带来的异常影响，也可以减少 API Server 监控 ConfigMap 的变化所带来的性能损耗（由于明确知道了配置绝对不会变，各个节点上的 Kubelet 就不再需要与集群的核心大脑（API Server）建立长连接去时刻轮询监听文件变化。在拥有数千个 Pod 的大规模生产集群中，这能极大地降低 API Server 的 CPU 和网络负载。）。



### Secret：捍卫核心资产的信息安全底线

在任何实际的业务架构中，我们都必须保护服务之间的连接凭证（如数据库密码、API Token）。为了确保用户数据不被泄露，K8s 提供了 **Secret** 资源专门用于管理机密数据。

#### 物理级别的内存隔离 (tmpfs)

Secret 的工作机制与 ConfigMap 类似，但它在底层的挂载方式上有着极其精妙的安全设计。

当 K8s 将 Secret 作为 Volume 挂载到运行的容器中时，它并不是写在宿主机的物理磁盘上。相反，K8s 会创建一个基于 RAM 的 **tmpfs（临时文件系统）**。这意味着，哪怕是极其罕见的极端情况——黑客物理入侵了数据中心的服务器并拔走了硬盘，他们也无法从中提取出 Secret 数据，因为系统断电的瞬间，内存中的机密信息就彻底灰飞烟灭了。

#### 进阶安全：从静态加密到传输层防护

这里必须要澄清一个常见的误区：**Secret 默认仅仅是 Base64 编码，而不是加密。** 在 K8s 的核心数据库 etcd 中，默认情况下 Secret 是明文存储的。

如果你处在对合规性要求极高的组织中，仅仅依赖 tmpfs 是不够的。完整的架构安全体系必须涵盖以下几个维度：

1. **静态数据加密 (Encryption at Rest)：** 必须在 K8s 中集成 KMS（密钥管理系统），让 API Server 在将 Secret 写入 etcd 之前，先使用 KMS 提供的密钥进行高强度加密。
    
2. **传输层通信安全 (Encryption in Transit)：** 保护 etcd 免受直接的物理或网络层攻击只是一方面。我们在处理系统架构的信息安全时，必须建立全链路的安全模型。即使 etcd 静态数据加密了，客户端与 Kubernetes API Server 之间的通信如果被窃听依然是致命的。因此，强制要求高强度的传输层加密（例如配置仅允许 **TLS 1.3** 握手协议，杜绝降级攻击）是企业级集群不可妥协的底线要求。
3. **拥抱 CSI 驱动体系：** 对于真正的大型组织，最好的做法是根本不把敏感凭证存在 K8s 里。通过引入 CSI（容器存储接口）驱动程序，我们可以直接将外部企业级金库（如 AWS Secrets Manager 或 HashiCorp Vault）中的机密数据，安全、实时地映射到 Pod 中。

只有在配置与安全这两个底层基石完全夯实之后，我们才敢放心地让承载数据的“重型装甲”——数据库，正式驶入战场。







使用 Secret 管理认证
	1. 在任何实际的应用程序中，我们都需要保护服务之间的连接。
	2. 为了确保用户及其数据的安全
	3. Kubernetes提供了内置的Secret资源来管理机密数据。
	4. Volume实际上是一个文件或目录，可以将用户指定的位置挂载到正在运行的容器中。对于Secret，Volume被创建为基于RAM的tmpfs文件系统，然后挂载到容器中。这样一来，即使机器被物理上入侵（在云中是不太可能的，但在数据中心是可能的），攻击者也很难获取到Secret。
	5. 默认情况下，Secret是未加密存储的。如果你想要加密存储Secret，可以集成密钥提供程序以获取一个密钥，Kubernetes将使用该密钥来加密集群中的所有Secret。
	6. 请注意，尽管这样可以保护密钥免受对etcd数据库的直接攻击的影响，但仍需确保Kubernetes API服务器的访问安全。
	7. CSI（Container Storage Interface，容器存储接口）驱动程序使你能够使用位于Kubernetes集群外部的KMS（Key ManagementSystem，密钥管理系统）。这通常是大型组织或受监管组织内合规性和安全性的要求
