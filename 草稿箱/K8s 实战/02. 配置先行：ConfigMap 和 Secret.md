
## ConfigMap

1. 一切皆容器的云原生时代，应用部署的最佳实践是先将应用打包成镜像，再分发至不同的环境。为了在开发、测试、生产环境中复用同一个版本的镜像（可移植性），我们绝不能将配置文件“写死”并打包进镜像中。为此，Kubernetes 设计了 ConfigMap 资源对象来实现应用配置的统一管理。
2. ConfigMap 供容器使用的典型用法：
	- 生成容器内的环境变量，适合简单的配置项（如数据库连接地址、环境标识）。
	- 设置容器启动命令的命令行参数（需设置为环境变量）
	- 以 Volume 的形式挂载为容器的文件或目录，适合复杂的长文本配置（如自定义的 `nginx.conf` ）。
3. ConfigMap 必须在 Pod 之前创建，Pod 才能引用它。如果 ConfigMap 不存在，或者被 Pod 引用的 key 不存在，则 Pod 无法启动。但是，如果在 Pod 的配置中设置引用 ConfigMap 是可选的（optional），则 Pod 会被正常创建和启动，只是容器内引用的 ConfigMap 的环境变量的值或挂载的文件的内容会被设置为空。
4. 如果 Pod 使用 envFrom 基于 ConfigMap 定义环境变量，则无效的环境变量名称（例如以数字开头）将被忽略，并在事件中被记录为 “InvaildVariableNames”。
5. ConfigMap 受命名空间限制，只有相同的命名空间中的 Pod 才可以引用它。
6. ConfigMap 无法用于静态 Pod。
7. 如果需要为容器传递命令行参数，可以使用$（ENV_KEY）插值语法来获取相应的环境变量。
8. 创建 ConfigMap 的三种方式：
	- 基于本地配置文件
	- 基于 kubectl 命令行  `kubectl create configmap NAME --from-file=[key=]source`
	- 基于 kustomize
9. ConfigMap 存在局限性：
	- 直接更改 ConfigMap 资源，**不会**自动触发现有 Pod 的滚动更新。虽然你可以在应用程序内自己编写监听文件变化的逻辑（例如 Go 的 viper 包）来实现热更新，但强烈不推荐。因为这会绕过 K8s 的健康检查（探针 Probes）机制，假设热更新了一个错误的数据库连接串，连不上数据库，应用程序会疯狂给用户返回 HTTP 500 错误。因为在 K8s 眼里，这个 Pod 的“生命周期”根本没有发生改变，它依然是那个处于 `Running` 状态的旧 Pod，K8s 并没有触发“发布”流程，自然也不会用探针去拦截它。等探针察觉到不对劲把 Pod 踢下线时，错误已经暴露给真实用户了。也许你会说，那我就不实现热更新了，直接改线上的 ConfigMap，那更加可怕，你这个改错的配置很可能在 3 天后， Pod 发生一次意外的崩溃被 K8s 重新调度，因而读取了你 3 天前的错误配置，起不来了。而你一脸懵，为什么 3 天前改的配置今天凌晨引爆了系统。这种不可控的随机切换完全可以避免。
	- ConfigMap 会失忆。ConfigMap 无法做到像 Deployment 那样，修改后发现不对能够立马回滚。K8s 会为 Deployment 在底层保留上一个版本的记录（也就是 ReplicaSet），而 **不会保存** ConfigMap 的历史版本。因此，ConfigMap 回滚极其困难，
10. 为什么必须坚持“配置不可变、版本化发布”？
	- 为了解决这个痛点，我们需要引入“不可变基础设施”的思维：在 ConfigMap 的命名中加入版本号（例如 `web-config-v1`）。
	- 当配置发生变更时，**绝对不要**去修改现有的 `web-config-v1`。而是创建一个全新的 `web-config-v2`，随后修改 Deployment 的 YAML 文件，将引用的 ConfigMap 指向 v2。
	- 这种做法会立刻触发 Deployment 的滚动更新机制。K8s 会先启动挂载了 v2 配置的新 Pod，执行健康检查，确认无误后再优雅停机旧 Pod。如果发现 v2 配置有误引发了故障，由于集群中依然保留着完整的 `web-config-v1`，你只需一条命令即可将 Deployment 瞬间回滚。
	- 这就是 ConfigMap **标准的、安全的更新流程（利用版本化 ConfigMap + 触发 Deployment 滚动更新）**
11. K8s 官方显然也深刻意识到了“原位修改配置”带来的灾难。因此，从 Kubernetes v1.19 版本开始，为 ConfigMap 新增了 immutable 字段，用于设置配置数据不可修改，如需修改，只能通过先删除再重建来实现。这个特性可以完美地配合实现我们的版本化 ConfigMap。这么做的好处，不仅消除了意外更新 ConfigMap 带来的异常影响，也可以减少 API Server 监控 ConfigMap 的变化所带来的性能损耗（由于明确知道了配置绝对不会变，各个节点上的 Kubelet 就不再需要与集群的核心大脑（API Server）建立长连接去时刻轮询监听文件变化。在拥有数千个 Pod 的大规模生产集群中，这能极大地降低 API Server 的 CPU 和网络负载。）。



### Secret：捍卫核心资产的信息安全底线

在任何实际的业务架构中，我们都必须保护服务之间的连接凭证（如数据库密码、API Token）。为了确保用户数据不被泄露，K8s 提供了 **Secret** 资源专门用于管理机密数据。

1. Secret中的数据应该以容易隐藏的方式进行存储和处理，并尽可能地进行静态加密。Secret数据是以base64编码的方式展现的，它们并没有被加密，理解这一点至关重要。Secret一旦被注入Pod中，Pod便可以以明文的方式查看机密数据。
2. Secret应该只存储少量数据，默认情况下，Kubernetes限制了base64编码之后的数据最多不超过1MB（和 ConfigMap 一样，这是由底层 etcd 数据库的限制决定的，不能用来存大文件），考虑到编码的额外开销，Base64 编码会带来约 33% 的体积膨胀，请确保实际的机密数据不超过750KB。
3. Secret 和 ConfigMap 在使用方式上没有任何区别，ConfigMap 的创建方法、注入 Pod 方法，命名空间限制，局限性，不可变特性，版本化管理等都一模一样，它们之间的根本区别在于底层存储和安全机制。挂载进入 Pod 之后，ConfigMap 会被真实的存储在 Pod 所在节点的物理磁盘上，Secret 不会被写入物理磁盘，而是挂载进一个专门划出来的 RAM 临时文件系统（**`tmpfs` / 内存盘**），Pod 消失 Secret 即消失。
4. Secret 在 etcd 里面需要“静态加密”，K8s 允许通过配置 `EncryptionConfiguration` 或接入外部 KMS，在 API Server 将数据写入 etcd 之前将其**高强度加密**。
5. Secret 和 ConfigMap 是两种资源，因而可以分开配置 RBAC 规则，确保密码等敏感数据不会被普通开发人员读写。
6. 当 K8s 将 Secret 作为 Volume 挂载到运行的容器中时，它并不是写在宿主机的物理磁盘上。相反，K8s 会创建一个基于 RAM 的 **tmpfs（临时文件系统）**。这意味着，哪怕是极其罕见的极端情况——黑客物理入侵了数据中心的服务器并拔走了硬盘，他们也无法从中提取出 Secret 数据，因为系统断电的瞬间，内存中的机密信息就彻底灰飞烟灭了。
7. 机密数据先是通过网络到达 **Kubelet 的私有程序内存**，然后被 Kubelet 写入到了伪装成文件系统的 **RAM 内存盘 (tmpfs)** 中，最后挂载给了容器。
8. 在对合规性要求极高的组织中，仅仅依赖 tmpfs 是不够的。完整的架构安全体系必须包含以下几个维度：
	- 必须在 K8s 中集成 KMS（密钥管理系统）
	- 必须使用 HTTPS/TLS 加密传输
	- （大型组织）可引入 CSI（容器存储接口）驱动程序


### 配置挂载的底层原理

1. **获取资源**：Kubelet 监听到新 Pod 调度后，通过 mTLS 安全通道向 API Server 拉取所需的 ConfigMap 和 Secret 数据，暂存于自身进程的内存中。
2. **底层落盘**：Kubelet 在宿主机为资源“安家”，将 ConfigMap 写入真实的物理磁盘目录，而为 Secret 向 Linux 内核申请一块基于 RAM 的 `tmpfs`（临时内存盘）以确保阅后即焚。
3. **原子投射**：为了支持安全的热更新，Kubelet 不会直接暴露源文件，而是将内容写入带时间戳的隐藏目录，并对外暴露一个指向它的软链接（Symlink）。
4. **绑定映射**：一切就绪后，Kubelet 指令底层容器运行时（如 containerd），利用 Linux 的 `bind mount` 技术，将宿主机上的软链接目录精准透传到容器指定的路径下。
